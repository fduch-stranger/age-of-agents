# Codex Context And Building Attribution

- Codex `response_item` `function_call` records may split tool identity across `payload.namespace` and `payload.name` (e.g. `namespace: "mcp__serena"`, `name: "initial_instructions"`). `packages/server/src/sources/codex.ts` must combine these before canonicalization so MCP tools become `mcp__serena__initial_instructions` and hit the shared `mcp__* -> guild` mapping. For `multi_agent_v1`, `spawn_agent` normalizes to `Workflow`/barracks, while `wait_agent`, `send_input`, and `close_agent` stay qualified (`mcp__multi_agent_v1__...`) and go to guild via the shared MCP prefix rule.
- Codex `event_msg.token_count.info.model_context_window` is model capacity, not current context usage. `packages/server/src/sources/codex.ts` should set `usage-total.context` from positive `last_token_usage.input_tokens` and keep capacity separate as optional `contextWindow`.
- Codex `type: "compacted"` records should not emit `turn-end`; compaction is a context-window event and real Codex sessions can continue work immediately afterward.
- State/social building attribution is centralized in `packages/shared/src/index.ts`: `homeBuildingForTheme`, `awaitingBuildingForTheme`, `completedBuildingForTheme`, `recoveryBuildingForTheme`, `activityBuildingForHero`, and `activityBuildingForAction`. Client wrappers remain in `packages/client/src/game/home-building.ts` for existing imports.
- Codex lifecycle normalization (2026-06-20): `task_started` emits `thinking`; `task_complete`/`turn_complete` emit `turn-end` and route live heroes to the completed social building (`garden`/`hydroponics`); `turn_aborted` emits `turn-aborted`, puts heroes in `recovering`, and routes them to recovery (`shrine`/`medbay`). Idle/sleeping heroes return to a stable social home distributed by project+session id. Tool rules remain separate from social placement.
- Historical Codex stats in `packages/server/src/building-stats.ts` credit output deltas after `task_complete`/`turn_complete` to both theme resting buildings (`garden`, `hydroponics`) and after `turn_aborted` to both recovery buildings (`shrine`, `medbay`), because `/building-stats` is not theme-aware. Codex usually writes the final `token_count` before the lifecycle event, so the scanner keeps one pending output delta and redirects it when the next event is completion/abort; do not eagerly accumulate Codex token deltas without preserving that ordering.
